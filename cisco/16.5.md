## Topologia

```
PC-A ──────── S1 ──────── R1
(192.168.1.3) (VLAN1:     (G0/0/1:
               192.168.1.11) 192.168.1.1)
```

> [!NOTE]
> La topologia è composta da tre dispositivi in un unico segmento di rete `/24`.
> Il PC-A si connette allo switch S1 tramite cavo Ethernet; S1 è collegato a R1 tramite la porta G0/0/1.
> Il cavo console azzurro collegato direttamente tra PC-A e la porta **Console** di R1 (e di S1) è il canale di configurazione iniziale: è l'unico accesso disponibile prima di configurare qualsiasi servizio remoto.

## Tabella di indirizzamento

| Dispositivo | Interfaccia | Indirizzo IP  | Subnet Mask   | Gateway Predefinito |
| ----------- | ----------- | ------------- | ------------- | ------------------- |
| R1          | G0/0/1      | 192.168.1.1   | 255.255.255.0 | N/A                 |
| S1          | VLAN 1      | 192.168.1.11  | 255.255.255.0 | 192.168.1.1         |
| PC-A        | NIC         | 192.168.1.3   | 255.255.255.0 | 192.168.1.1         |

> [!NOTE]
> Il router **non ha gateway predefinito** perché è lui stesso il gateway della rete.
> Lo switch ha una SVI (Switch Virtual Interface) sulla VLAN 1 per permettere la gestione remota; senza di essa non sarebbe raggiungibile via SSH o Telnet.

## Introduzione e scenario

In questo laboratorio si configurano R1 e S1 per accettare sessioni **SSH** per la gestione remota, sostituendo Telnet (non sicuro). Si applicano inoltre misure di sicurezza comunemente raccomandate come best-practice: password robuste, cifratura, banner di avviso, timeout di sessione e blocco accessi dopo tentativi falliti.

> [!WARNING]
> **Telnet trasmette tutto in chiaro**, incluse le password. Chiunque abbia accesso al segmento di rete può catturare le credenziali con strumenti come Wireshark in pochi secondi. SSH cifra l'intero canale: è il protocollo standard per la gestione remota di dispositivi Cisco.

> [!NOTE]
> I router utilizzati sono Cisco 4221 con Cisco IOS XE Release 16.9.4 (immagine universalk9). Gli switch sono Cisco Catalyst 2960s con Cisco IOS Release 15.2(2). Output e comandi potrebbero variare leggermente con altri modelli o versioni IOS.

## Parte 1: Configurare le impostazioni di base del dispositivo

### Fase 1: Cablare la rete come illustrato nella topologia

Collegare i dispositivi secondo la tabella di indirizzamento. Verificare che:
- Il cavo console sia collegato tra PC-A e la porta Console di R1
- S1 sia collegato a R1 sulla porta G0/0/1
- PC-A sia collegato a S1 su una porta di accesso

> [!TIP]
> In Packet Tracer, usa il cavo **azzurro** (console) tra PC e router/switch per la configurazione iniziale. Usa il cavo **copper straight-through** (nero) per le connessioni PC–switch e switch–router.

### Fase 2: Inizializzare e ricaricare il router e lo switch

Prima di iniziare, assicurarsi che i dispositivi non abbiano configurazioni residue.

**Su R1 e S1:**
```
Router> enable
Router# erase startup-config
Router# reload
```

> [!WARNING]
> Il comando `erase startup-config` cancella la configurazione salvata nella NVRAM. Dopo il `reload`, il dispositivo si avvia senza nessuna configurazione: chiederà se si vuole avviare la procedura guidata di configurazione (Setup Mode) — rispondere **no**.

### Fase 3: Configurare il router R1

> **Aprire la finestra di configurazione** (connettiti via cavo console a R1)

#### a. Accesso alla modalità privilegiata

```
Router> enable
Router#
```

> [!NOTE]
> Il prompt `Router>` indica la **modalità utente** (EXEC utente): accesso limitato, solo comandi di visualizzazione.  
> Il prompt `Router#` indica la **modalità privilegiata** (EXEC privilegiato): accesso completo a tutti i comandi di verifica e configurazione.  
> Il comando `enable` eleva i privilegi dalla modalità utente a quella privilegiata.

---

#### b. Assegnare il nome al dispositivo

```
Router# configure terminal
Router(config)# hostname R1
R1(config)#
```

> [!NOTE]
> `configure terminal` entra in **modalità configurazione globale**: da qui si modificano le impostazioni che riguardano l'intero dispositivo.  
> `hostname R1` imposta il nome del router. Il prompt cambia immediatamente da `Router` a `R1`.  
> Il nome host è obbligatorio anche per generare correttamente le chiavi RSA per SSH (il nome viene incorporato nell'identificatore della chiave).

---

#### c. Disabilitare la ricerca DNS

```
R1(config)# no ip domain-lookup
```

> [!TIP]
> Senza questo comando, ogni volta che si digita un comando errato (es. `conft` invece di `conf t`), il router interpreta la stringa come un nome host e tenta una risoluzione DNS — bloccandosi per diversi secondi.  
> `no ip domain-lookup` disabilita questo comportamento. È buona norma applicarlo **sempre all'inizio** di ogni sessione di configurazione in laboratorio.  
> Se il router si blocca in una ricerca DNS, premi **CTRL+SHIFT+6** per interromperla.

---

#### d. Assegnare la password privileged EXEC cifrata

```
R1(config)# enable secret class
```

> [!NOTE]
> `enable secret` imposta la password richiesta per entrare in modalità privilegiata (`Router#`).  
> A differenza di `enable password`, il comando `enable secret` cifra la password con **MD5**: anche se un malintenzionato leggesse il file di configurazione, non troverebbe la password in chiaro.  
> **Regola fondamentale:** usa sempre `enable secret`, mai `enable password` da solo.

> [!WARNING]
> Se nel file di configurazione sono presenti sia `enable secret` che `enable password`, IOS usa **sempre** `enable secret` e ignora `enable password`. In questo lab usiamo solo `enable secret`.

---

#### e. Impostare la password di console

```
R1(config)# line console 0
R1(config-line)# password cisco
R1(config-line)# login
R1(config-line)# exit
```

> [!NOTE]
> `line console 0` seleziona la linea di connessione fisica via cavo console. Il numero `0` indica la prima (e di solito unica) porta console del dispositivo.  
> `password cisco` imposta la password per questa linea.  
> `login` **attiva la richiesta di password** al momento della connessione: senza questo comando, la password impostata verrebbe ignorata e chiunque potrebbe accedere senza autenticarsi.

---

#### f. Impostare la password VTY

```
R1(config)# line vty 0 4
R1(config-line)# password cisco
R1(config-line)# login
R1(config-line)# exit
```

> [!NOTE]
> Le linee **VTY (Virtual TeleType)** sono connessioni remote virtuali: rappresentano le sessioni Telnet/SSH simultanee che il dispositivo può accettare.  
> `line vty 0 4` configura le prime 5 linee VTY (da 0 a 4), consentendo fino a 5 connessioni remote simultanee.  
> In questa fase iniziale si usa ancora `login` (con password semplice): nella Parte 2 si sostituirà con `login local` e l'autenticazione tramite database utenti locali.

> [!WARNING]
> La password `cisco` è usata qui solo come configurazione di base temporanea. Nella Parte 2 verrà sostituita con password più robuste conformi ai requisiti di sicurezza del lab.

---

#### g. Creare un banner di avviso (MOTD)

```
R1(config)# banner motd #
Accesso autorizzato esclusivamente al personale autorizzato.
Qualsiasi accesso non autorizzato e' vietato e perseguibile.
#
```

> [!NOTE]
> **MOTD = Message Of The Day.** Il banner viene visualizzato a chiunque tenti di connettersi al dispositivo, prima dell'autenticazione.  
> Il carattere `#` è il **delimitatore**: IOS registra tutto ciò che si digita fino al secondo `#` come testo del banner. Si può usare qualsiasi carattere come delimitatore, purché non compaia nel testo del messaggio.

> [!TIP]
> Il banner MOTD ha valore legale: avvisa gli utenti non autorizzati che l'accesso è vietato. In ambienti reali, i messaggi di avviso sono spesso obbligatori per rendere perseguibili gli accessi non autorizzati. **Non scrivere mai** banner che invitano all'accesso (es. "Benvenuto!") — potrebbero essere usati come difesa da un intruso.

---

#### h. Configurare e attivare l'interfaccia G0/0/1

```
R1(config)# interface GigabitEthernet0/0/1
R1(config-if)# ip address 192.168.1.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit
```

> [!NOTE]
> `interface GigabitEthernet0/0/1` (abbreviabile in `int g0/0/1`) entra in modalità configurazione interfaccia.  
> `ip address 192.168.1.1 255.255.255.0` assegna l'indirizzo IP e la subnet mask.  
> `no shutdown` **attiva l'interfaccia**: per impostazione predefinita, le interfacce dei router sono in stato `administratively down` e vanno abilitate esplicitamente. Senza questo comando, l'interfaccia non trasmetterà traffico.

> [!TIP]
> Verifica lo stato delle interfacce con:
> ```
> R1# show ip interface brief
> ```
> L'output mostra indirizzo IP, stato fisico (`up`/`down`) e stato protocollo (`up`/`down`). Entrambi devono essere `up` per una connessione funzionante.

---

#### i. (Switch S1) Configurare la SVI sulla VLAN 1

Connettiti ora via console a **S1** e ripeti i passi a–g con hostname `S1`, poi configura la SVI:

```
S1(config)# interface vlan 1
S1(config-if)# ip address 192.168.1.11 255.255.255.0
S1(config-if)# no shutdown
S1(config-if)# exit
S1(config)# ip default-gateway 192.168.1.1
```

> [!NOTE]
> Gli switch di livello 2 non instradano pacchetti, ma possono essere gestiti remotamente tramite una **SVI (Switch Virtual Interface)** — un'interfaccia logica associata a una VLAN.  
> `ip default-gateway 192.168.1.1` indica allo switch dove inviare il traffico destinato a reti esterne (es. una connessione SSH proveniente da un'altra subnet). Senza gateway, lo switch non è raggiungibile da fuori la propria rete locale.

---

#### j. Salvare la configurazione

```
R1# write memory
```
oppure equivalente:
```
R1# copy running-config startup-config
```

> [!NOTE]
> La **running-config** è la configurazione attiva in RAM: va persa al riavvio.  
> La **startup-config** è la configurazione salvata in NVRAM: viene caricata all'avvio.  
> `write memory` (abbreviabile in `wr`) copia la running-config nella startup-config. **Salva sempre** dopo ogni modifica importante.

> [!WARNING]
> Un errore frequente in laboratorio è dimenticare di salvare: se il router viene riavviato accidentalmente, tutta la configurazione va persa. Prendi l'abitudine di salvare dopo ogni fase completata.

### Fase 4: Configurare PC-A

> **Aprire il prompt dei comandi sul PC-A**

#### a–b. Impostare indirizzo IP e gateway

Configura il PC con:
- **IP:** `192.168.1.3`
- **Subnet mask:** `255.255.255.0`
- **Gateway:** `192.168.1.1`

In Packet Tracer: clicca sul PC → scheda **Desktop** → **IP Configuration** → seleziona **Static** e inserisci i valori.

### Fase 5: Verificare la connettività di rete

> **Aprire la finestra di configurazione su PC-A**

Dal terminale di PC-A:

```
PC> ping 192.168.1.1
PC> ping 192.168.1.11
```

**Output atteso (ping riuscito):**
```
Pinging 192.168.1.1 with 32 bytes of data:
Reply from 192.168.1.1: bytes=32 time<1ms TTL=255
Reply from 192.168.1.1: bytes=32 time<1ms TTL=255
...
```

> [!TIP]
> Se il ping non riesce, controlla in ordine:
> 1. L'indirizzo IP e il gateway di PC-A sono corretti?
> 2. L'interfaccia G0/0/1 di R1 è attiva? (`show ip interface brief`)
> 3. La SVI VLAN 1 di S1 è attiva? (`show ip interface brief` su S1)
> 4. I cavi fisici sono collegati correttamente?

> [!NOTE]
> Il primo ping potrebbe fallire con "Request timed out": è normale in Packet Tracer. Si tratta della risoluzione ARP (Address Resolution Protocol) che avviene al primo pacchetto. Dal secondo ping in poi la risposta sarà positiva.

## Parte 2: Configurare le misure di sicurezza di base nel router

> **Aprire la finestra di configurazione su R1 (via console)**

### Fase 1: Configurare le misure di sicurezza

#### a. Cifrare tutte le password in chiaro

```
R1(config)# service password-encryption
```

> [!NOTE]
> Questo comando cifra tutte le password attualmente in chiaro nella running-config (console, VTY, `enable password`) usando la **cifratura di tipo 7** (algoritmo proprietario Cisco, reversibile).  
> Le password impostate con `enable secret` usano invece MD5 (tipo 5) e **non** vengono influenzate da questo comando — sono già cifrate in modo più robusto.

> [!WARNING]
> La cifratura Type 7 è **debole**: esistono tool online che la decodificano in pochi secondi. Non fare affidamento su di essa come unica protezione. Serve principalmente a impedire che qualcuno guardi lo schermo e legga la password, non a proteggerti da un attaccante determinato.  
> Per le password critiche usa sempre `enable secret` o `username … secret`.

---

#### b. Richiedere una password di minimo 12 caratteri

```
R1(config)# security passwords min-length 12
```

> [!NOTE]
> Questo comando impone una lunghezza minima per tutte le password future configurate sul dispositivo. Se si tenta di impostare una password più corta, IOS la rifiuterà con un messaggio di errore.  
> **Attenzione:** il comando ha effetto solo sulle password impostate **dopo** la sua configurazione. Le password già esistenti (es. `cisco` configurata nella Parte 1) non vengono invalidate automaticamente — per questo motivo il passo successivo le sostituisce.

---

#### c. Aggiornare le password per soddisfare il requisito di lunghezza

##### 1. Password privileged EXEC

```
R1(config)# enable secret $cisco!PRIV*
```

> [!NOTE]
> La password `$cisco!PRIV*` ha 12 caratteri e include lettere, numeri e simboli speciali — rispetta il requisito di `security passwords min-length 12`.  
> Questo comando sovrascrive la precedente `enable secret class` configurata nella Parte 1.

##### 2. Password di console

```
R1(config)# line console 0
R1(config-line)# password $cisco!!CON*
R1(config-line)# exit
```

##### 3. Password VTY

```
R1(config)# line vty 0 4
R1(config-line)# password $cisco!!VTY*
R1(config-line)# exit
```

> [!TIP]
> Dopo aver cambiato le password, prova subito ad aprire una nuova connessione console per verificare che la nuova password funzioni correttamente, **prima** di chiudere la sessione attiva. Se commetti un errore e non riesci più ad accedere, saresti bloccato fuori dal dispositivo.

---

#### d. Configurare SSH e accettare solo connessioni SSH remote

> [!NOTE]
> SSH (Secure Shell) richiede tre prerequisiti obbligatori prima di funzionare:
> 1. Un **hostname** diverso da "Router" (già configurato)
> 2. Un **domain name** (necessario per costruire l'identificatore della chiave RSA)
> 3. Una coppia di **chiavi crittografiche RSA** (cuore del protocollo SSH)

##### 1. Creare l'utente amministratore

```
R1(config)# username SSHadmin secret 55HadM!n2020
```

> [!NOTE]
> `username SSHadmin secret 55HadM!n2020` crea un account locale con nome `SSHadmin` e password cifrata con MD5.  
> Usa sempre `secret` (non `password`) per cifrare le credenziali degli utenti locali.  
> Questo account verrà usato per autenticarsi via SSH: il nome utente è **case sensitive** (`SSHadmin` è diverso da `sshadmin`).

##### 2. Impostare il domain name

```
R1(config)# ip domain-name ccna-lab.com
```

> [!NOTE]
> Il domain name è obbligatorio per la generazione delle chiavi RSA. IOS costruisce l'identificatore della chiave concatenando hostname e domain name: in questo caso sarà `R1.ccna-lab.com`.  
> Senza `ip domain-name`, il comando `crypto key generate rsa` restituirà l'errore:  
> `% Please define a domain-name first.`

##### 3. Generare le chiavi RSA (1024 bit)

```
R1(config)# crypto key generate rsa
```

Quando IOS chiede la dimensione del modulo, inserire:
```
How many bits in the modulus [512]: 1024
```

**Output atteso:**
```
% Generating 1024 bit RSA keys, keys will be non-exportable...
[OK] (elapsed time was 1 seconds)
```

> [!NOTE]
> Le **chiavi RSA** sono il fondamento crittografico di SSH: il router genera una coppia di chiavi (pubblica + privata). La chiave pubblica viene condivisa con il client all'apertura della connessione; la chiave privata rimane sul router e non viene mai trasmessa.  
> Il modulo da **1024 bit** è il valore richiesto da questo laboratorio.

> [!WARNING]
> SSH versione 2 richiede chiavi di **almeno 768 bit**. Con un modulo inferiore a 768 bit, il comando `ip ssh version 2` (che vedremo a breve) restituirà un errore. Con 1024 bit siamo abbondantemente nel range supportato.

##### 4. Forzare SSH versione 2

```
R1(config)# ip ssh version 2
```

> [!NOTE]
> SSH esiste in due versioni: SSH v1 (obsoleta, vulnerabile) e SSH v2 (sicura, raccomandata).  
> Questo comando forza il router ad accettare **solo** connessioni SSH v2, rifiutando le v1.  
> Quasi tutti i client moderni (incluso Packet Tracer) usano SSH v2 per impostazione predefinita.

##### 5. Configurare le VTY per accettare solo SSH

```
R1(config)# line vty 0 4
R1(config-line)# transport input ssh
R1(config-line)# login local
R1(config-line)# exit
```

> [!NOTE]
> **`transport input ssh`** — accetta **solo** connessioni SSH sulle linee VTY. Qualsiasi tentativo Telnet verrà rifiutato immediatamente.
>
> **`login local`** — usa il **database utenti locale** del router per l'autenticazione (l'account `SSHadmin` creato sopra). Questo è diverso da `login`, che usa la password impostata direttamente sulla linea VTY.
>
> | Comando | Comportamento |
> |---------|--------------|
> | `login` | Usa la `password` configurata sulla linea VTY |
> | `login local` | Usa il database `username`/`secret` del router |
>
> Per SSH con username, è **obbligatorio** usare `login local`.

---

#### e. Configurare sicurezza e best practice su console e VTY

##### 1. Timeout di inattività (5 minuti)

```
R1(config)# line console 0
R1(config-line)# exec-timeout 5 0
R1(config-line)# exit

R1(config)# line vty 0 4
R1(config-line)# exec-timeout 5 0
R1(config-line)# exit
```

> [!NOTE]
> `exec-timeout 5 0` disconnette automaticamente la sessione dopo **5 minuti** (5 minuti, 0 secondi) di inattività.  
> Senza questo comando, una sessione lasciata aperta (es. un terminale console dimenticato acceso) rimane accessibile indefinitamente — chiunque si avvicini può prendere il controllo del dispositivo senza dover inserire password.

> [!TIP]
> Il formato è `exec-timeout <minuti> <secondi>`.  
> `exec-timeout 0 0` disabilita completamente il timeout (non usarlo in produzione!).  
> `exec-timeout 5 30` imposta il timeout a 5 minuti e 30 secondi.

##### 2. Blocco accessi dopo tentativi falliti

```
R1(config)# login block-for 120 attempts 3 within 60
```

> [!NOTE]
> Questo comando attiva una protezione contro gli attacchi **brute-force**:
> - Se si verificano **3 tentativi di login falliti** (`attempts 3`)
> - entro **60 secondi** (`within 60`)
> - il router blocca **tutti** i nuovi tentativi di login per **120 secondi** (`block-for 120`)
>
> Durante il periodo di blocco, il router entra in **Quiet Mode**: non accetta connessioni di accesso remoto. Questo rende gli attacchi a forza bruta praticamente impraticabili.

> [!TIP]
> Per verificare lo stato del blocco accessi usa:
> ```
> R1# show login
> ```
> Il comando mostra se il router è in Quiet Mode, quanti secondi mancano alla fine del blocco e quanti tentativi falliti sono stati registrati.

> **Chiudere la finestra di configurazione**

## Parte 3: Verificare le misure di sicurezza nel router

> **Aprire la finestra di configurazione**

### Fase 1: Verificare che le porte non utilizzate siano disabilitate

```
R1# show ip interface brief
```

> [!NOTE]
> Le porte dei router sono **disabilitate per impostazione predefinita** (`administratively down`).  
> Verifica che tutte le interfacce non in uso si trovino in stato `administratively down`. Se qualche interfaccia non utilizzata risultasse `down` (non `administratively down`), significa che non è stata spenta esplicitamente — applicale il comando `shutdown` in modalità configurazione interfaccia.

**Esempio output atteso:**
```
Interface              IP-Address      OK? Method Status    Protocol
GigabitEthernet0/0/0   unassigned      YES unset  administratively down down
GigabitEthernet0/0/1   192.168.1.1     YES manual up                    up
```

Per disabilitare eventuali porte non in uso:
```
R1(config)# interface GigabitEthernet0/0/0
R1(config-if)# shutdown
R1(config-if)# exit
```

---

### Fase 2: Verificare le misure di sicurezza

#### a. Tentativo di connessione Telnet a R1

Dal terminale di PC-A:
```
PC> telnet 192.168.1.1
```

**Output atteso:**
```
Trying 192.168.1.1...
% Connection refused by remote host
```

> [!IMPORTANT]
> **Domanda:** R1 accetta la connessione Telnet? Spiegare.
>
> **Risposta:** No, R1 **rifiuta** la connessione Telnet. Questo è il comportamento corretto: nella Parte 2 abbiamo configurato `transport input ssh` sulle linee VTY, che accetta **esclusivamente** connessioni SSH. Il messaggio "Connection refused" conferma che Telnet è effettivamente bloccato.

---

#### b. Connessione SSH a R1

Dal terminale di PC-A:
```
PC> ssh -l SSHadmin 192.168.1.1
```

Inserire la password quando richiesta: `55HadM!n2020`

**Output atteso (connessione riuscita):**
```
Password:
R1>
```

> [!IMPORTANT]
> **Domanda:** R1 accetta la connessione SSH?
>
> **Risposta:** Sì, R1 accetta la connessione SSH. Dopo aver inserito la password corretta, si accede alla **modalità utente** (`R1>`). Per accedere alla modalità privilegiata è necessario digitare `enable` e inserire la password `$cisco!PRIV*`.

> [!NOTE]
> **Sintassi del comando `ssh -l`:**
>
> | Parte | Significato |
> |-------|-------------|
> | `ssh` | Comando SSH del terminale PC in Packet Tracer |
> | `-l` | Flag *login*: specifica lo **username** (non la password) |
> | `SSHadmin` | Username configurato sul router (case sensitive!) |
> | `192.168.1.1` | Indirizzo IP dell'interfaccia raggiungibile del router |

---

#### c. Verifica del blocco accessi

Con una nuova sessione SSH, inserire intenzionalmente credenziali errate per 3 volte consecutive entro 60 secondi.

```
PC> ssh -l SSHadmin 192.168.1.1
Password: [password sbagliata]
Password: [password sbagliata]
Password: [password sbagliata]
```

> [!IMPORTANT]
> **Domanda:** Cosa è successo dopo che l'accesso non è riuscito la seconda/terza volta?
>
> **Risposta:** Dopo il terzo tentativo fallito entro 60 secondi, il router entra in **Quiet Mode** e blocca tutti i nuovi tentativi di login per 120 secondi. La connessione viene chiusa e non è possibile avviarne di nuove finché il periodo di blocco non è trascorso.

---

#### d. Visualizzare lo stato login durante il blocco

Dalla sessione console attiva su R1:
```
R1# show login
```

**Esempio di output durante il blocco:**
```
    A login delay of 1 seconds is applied.
    Quiet-Mode access list NOT configured.

  Router enabled to watch for login Attacks.
  If more than 3 login failures occur in 60 seconds or less,
  logins will be disabled for 120 seconds.

  Router presently in Quiet-Mode.
    Will remain in Quiet-Mode for 111 seconds.
    Denying logins from all sources.
```

> [!NOTE]
> Il campo **"Will remain in Quiet-Mode for X seconds"** indica il tempo rimanente prima che le connessioni remote vengano nuovamente accettate. Durante il Quiet-Mode, l'accesso tramite console fisica rimane sempre possibile (non viene mai bloccato).

---

#### e. Connessione SSH dopo la scadenza del blocco (120 secondi)

Attendere che il Quiet-Mode termini, poi connettersi nuovamente:

```
PC> ssh -l SSHadmin 192.168.1.1
Password: 55HadM!n2020
```

> [!IMPORTANT]
> **Domanda:** Una volta eseguita correttamente la connessione, cosa viene visualizzato?
>
> **Risposta:** Viene visualizzato il **banner MOTD** configurato nella Fase 3g, seguito dal prompt di login. Dopo l'autenticazione corretta si accede alla modalità utente `R1>`. Il banner conferma che la misura di sicurezza è attiva.

---

#### f. Accesso alla modalità privilegiata

Dalla sessione SSH attiva:
```
R1> enable
Password: $cisco!PRIV*
R1#
```

> [!IMPORTANT]
> **Domanda:** Se l'utente digita male questa password, viene disconnesso dalla sessione SSH dopo tre tentativi non riusciti entro 60 secondi?
>
> **Risposta:** Sì. Il meccanismo `login block-for` monitora i tentativi falliti **su tutte le linee di accesso**, incluso il comando `enable`. Tre tentativi errati entro 60 secondi attivano il Quiet-Mode, disconnettendo la sessione SSH e bloccando nuovi accessi per 120 secondi.

---

#### g. Visualizzare la running-config

```
R1# show running-config
```

> [!TIP]
> Analizza l'output per verificare:
> - `service password-encryption` è presente
> - `security passwords min-length 12` è presente
> - Le password VTY e console appaiono cifrate (stringa esadecimale preceduta da `7`)
> - `enable secret` mostra una stringa hash (preceduta da `5`) — non la password in chiaro
> - `ip domain-name ccna-lab.com` è presente
> - `username SSHadmin secret ...` è presente
> - `ip ssh version 2` è presente
> - Le linee VTY mostrano `transport input ssh` e `login local`
> - `login block-for 120 attempts 3 within 60` è presente
> - `exec-timeout 5 0` è presente sulle linee console e VTY

## Parte 4: Configurare le misure di sicurezza di base nello switch

### Fase 1: Configurare le misure di sicurezza su S1

> **Aprire la finestra di configurazione su S1 (via console)**

Le stesse misure applicate a R1 vengono ora replicate su S1. I comandi sono identici, tranne per la parte specifica delle porte switch.

#### a. Cifrare tutte le password in chiaro

```
S1(config)# service password-encryption
```

#### b. Richiedere una password di almeno 12 caratteri

```
S1(config)# security passwords min-length 12
```

#### c. Aggiornare le password

```
S1(config)# enable secret $cisco!PRIV*

S1(config)# line console 0
S1(config-line)# password $cisco!!CON*
S1(config-line)# exit

S1(config)# line vty 0 4
S1(config-line)# password $cisco!!VTY*
S1(config-line)# exit
```

#### d. Configurare SSH su S1

```
S1(config)# hostname S1
S1(config)# ip domain-name ccna-lab.com
S1(config)# username SSHadmin secret 55HadM!n2020
S1(config)# crypto key generate rsa
```
Inserire `1024` quando richiesto.

```
S1(config)# ip ssh version 2
S1(config)# line vty 0 4
S1(config-line)# transport input ssh
S1(config-line)# login local
S1(config-line)# exit
```

> [!NOTE]
> La procedura SSH sullo switch è **identica** a quella del router. Gli switch Catalyst gestiti con IOS supportano SSH nello stesso modo dei router.

#### e. Timeout e blocco accessi

```
S1(config)# line console 0
S1(config-line)# exec-timeout 5 0
S1(config-line)# exit

S1(config)# line vty 0 4
S1(config-line)# exec-timeout 5 0
S1(config-line)# exit

S1(config)# login block-for 120 attempts 3 within 60
```

#### f. Disabilitare tutte le porte inutilizzate

```
S1(config)# interface range FastEthernet0/2 - 24
S1(config-if-range)# shutdown
S1(config-if-range)# exit
S1(config)# interface range GigabitEthernet0/1 - 2
S1(config-if-range)# shutdown
S1(config-if-range)# exit
```

> [!NOTE]
> `interface range` permette di configurare **più interfacce contemporaneamente** con un singolo blocco di comandi.  
> Disabilitare le porte inutilizzate è una best-practice fondamentale: impedisce che qualcuno colleghi un dispositivo non autorizzato alla rete semplicemente inserendo un cavo in una porta libera.

> [!TIP]
> Prima di disabilitare le porte, verifica quali sono attivamente in uso con:
> ```
> S1# show ip interface brief
> ```
> oppure:
> ```
> S1# show interfaces status
> ```
> Disabilita solo le porte in stato `notconnect` o `disabled`. Non disabilitare mai le porte collegate a R1 o a PC-A!

---

### Fase 2: Verificare che le porte non utilizzate siano disabilitate

```
S1# show ip interface brief
```

**Output atteso:**
```
Interface     IP-Address   OK? Method Status             Protocol
Vlan1         192.168.1.11 YES manual up                 up
Fa0/1         unassigned   YES unset  up                 up        ← porta verso R1
Fa0/2         unassigned   YES unset  administratively down down   ← disabilitata

> [!NOTE]

## Risposte alle domande di approfondimento

### Domanda 1

**Il comando `password cisco` è stato immesso per la console e le linee VTY nella configurazione di base nella Parte 1. Dopo l'applicazione delle best-practice di sicurezza, quando viene utilizzata questa password?**

**Risposta:** La password `cisco` configurata nella Parte 1 **non viene più utilizzata** dopo le modifiche della Parte 2. Ci sono due motivi:

1. Nella Parte 2 le password sono state sostituite con `$cisco!!CON*` (console) e `$cisco!!VTY*` (VTY), che sono le password ora attive.
2. Le linee VTY sono state riconfigurare con `login local` anziché `login`: l'autenticazione non usa più la password della linea, ma il database utenti locale (account `SSHadmin`). La password VTY rimane nel file di configurazione ma viene di fatto ignorata fintantoché è attivo `login local`.

---

### Domanda 2

**Le password preconfigurate di lunghezza inferiore a 10 caratteri sono interessate dal comando `security passwords min-length 12`?**

**Risposta:** No. Il comando `security passwords min-length 12` ha effetto **solo sulle password impostate dopo la sua configurazione**. Le password già esistenti (es. `cisco` da 5 caratteri) rimangono valide fino a quando non vengono modificate. Questo è il motivo per cui la Fase 1c della Parte 2 richiede espressamente di **aggiornare tutte le password** dopo aver impostato la lunghezza minima: il sistema non invalida automaticamente quelle vecchie.
