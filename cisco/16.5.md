## 🏛️ Strategia di Difesa: I 3 Livelli di Accesso

Prima di iniziare, è fondamentale capire **cosa** stiamo proteggendo. In questa esercitazione configureremo tre tipi di "serrature" che convivono nello stesso dispositivo:

> [!NOTE]
> ### 1. Il Portone Fisico (Console)
> Rappresenta l'accesso locale. È il cavo azzurro collegato direttamente al router.
> * **Cosa permette:** È l'accesso di emergenza. Se la rete cade o l'SSH non funziona, la Console è l'unica via rimasta.
> * **Obiettivo:** Impedire che chiunque entri fisicamente nella stanza possa comandare il router.

> [!TIP]
> ### 2. Il Tunnel Remoto (SSH)
> Rappresenta l'accesso via rete. È come entrare in ufficio passando da internet.
> * **Cosa permette:** Permette di gestire il router comodamente dal proprio PC.
> * **Obiettivo:** Creare un tunnel cifrato e sicuro. Sostituisce il vecchio Telnet (insicuro) perché cripta tutto il traffico, incluse le password.

> [!CAUTION]
> ### 3. La Cassaforte (Enable Secret)
> Non è una porta, ma un **livello di privilegio**.
> * **Cosa permette:** Una volta entrati (sia da Console che da SSH), si è "utenti semplici" (`>`). Per diventare "Amministratori" (`#`) e cambiare le impostazioni, serve questa password.
> * **Obiettivo:** Anche se un ladro riuscisse a superare il Portone o il Tunnel, non potrebbe comunque rubare nulla senza la chiave della Cassaforte.

## Parte 1 — Fondamenta e Perimetro

### Cambio Nome (Hostname)

```bash
Router> enable
Router# configure terminal
Router(config)# hostname RTR-A
```

### Disattivazione Ricerca DNS

```bash
RTR-A(config)# no ip domain-lookup
```

> [!TIP]
> Senza questo comando, ogni volta che digiti male un comando il router lo scambia per un nome di dominio e si mette ad aspettare una risposta DNS — con un'attesa di circa 30 secondi prima di darti un errore. Con `no ip domain-lookup` l'errore è immediato.

## Parte 2 — Blindare il Router

### A — Sicurezza delle Password

**Lunghezza minima obbligatoria (10 caratteri):**

```bash
RTR-A(config)# security passwords min-length 10
```

**Protezione della "Cassaforte" (Privileged EXEC Mode):**

```bash
RTR-A(config)# enable secret @Sicurezza123!
```

**Protezione del "Portone" (Console):**

```bash
RTR-A(config)# line console 0
RTR-A(config-line)# password @Cons1234!
RTR-A(config-line)# login
RTR-A(config-line)# exit
```

> [!NOTE]
> `enable secret` cifra la password con MD5 ed è sempre preferibile a `enable password`, che la salva in chiaro. Non usare mai `enable password` in produzione.

---

### B — Configurazione SSH (Accesso Remoto Sicuro)

**Definire il dominio e creare l'utente:**

```bash
RTR-A(config)# ip domain-name lab.sicuro.it
RTR-A(config)# username SSHadmin secret @AdminPass123!
```

> [!NOTE]
> Il `domain-name` funziona come il "cognome" del router: viene combinato con l'hostname (`RTR-A.lab.sicuro.it`) per generare una chiave crittografica univoca. Senza di esso il comando `crypto key generate rsa` non funziona.

**Generare le chiavi crittografiche RSA:**

```bash
RTR-A(config)# crypto key generate rsa
```

> Quando richiesto, inserisci: `1024`

> [!TIP]
> Usiamo **1024 bit** invece di 512 perché è il minimo per abilitare SSH versione 2, più sicuro della versione 1. Più alto è il numero di bit, più robusto è il tunnel cifrato — ma anche più lento da generare.

**Attivare SSH sulle linee virtuali (VTY):**

```bash
RTR-A(config)# line vty 0 4
RTR-A(config-line)# transport input ssh
RTR-A(config-line)# login local
RTR-A(config-line)# exit
```

> [!TIP]
> `line vty 0 4` apre 5 sessioni remote simultanee (da 0 a 4). `transport input ssh` blocca Telnet — che invia tutto in chiaro, password inclusa — e permette solo connessioni SSH cifrate. `login local` dice al router di usare il database utenti locale (l'utente `SSHadmin` creato sopra).

---

### C — Blocco Attacchi Brute Force

```bash
RTR-A(config)# login block-for 120 attempts 3 within 60
```

> [!WARNING]
> Questo comando blocca **tutti** gli accessi per 120 secondi dopo 3 tentativi falliti in 60 secondi. Se ti blocchi fuori durante il lab, devi semplicemente aspettare 2 minuti prima di riprovare.

## Parte 3 — Il Test del Ladro (Verifica) 🕵️

Dal **Command Prompt** di un PC in Packet Tracer:

**1. Testa l'accesso SSH:**

```bash
PC> ssh -l SSHadmin 192.168.1.1
```

> Inserisci la password: `@AdminPass123!`
>
> *(Nota: `-l` è una L minuscola, sta per "login")*

**2. Verifica la cifratura delle password:**

```bash
RTR-A# show running-config
```

> [!NOTE]
> Controlla che le password non siano leggibili. Al posto di `@Sicurezza123!` vedrai una stringa cifrata tipo `$9$abc123...`. Se vedi la password in chiaro, hai dimenticato `service password-encryption`.

> [!CAUTION]
> Se provi a connetterti via Telnet invece di SSH, la connessione deve essere **rifiutata**. Se Telnet funziona ancora, significa che `transport input ssh` non è stato applicato correttamente — torna alla Parte 2B e ricontrolla.
>


**3. Prova l'accesso Telnet (test di sicurezza):**

```
C:\> telnet 192.168.1.1
```

Output atteso:

```
Trying 192.168.1.1 ...Open
[Connection to 192.168.1.1 closed by foreign host]
```

> [!IMPORTANT]
> Questo è **esattamente il comportamento corretto**. Non è un errore.
>
> Il messaggio `Open` seguito immediatamente da `closed by foreign host` significa che il router ha ricevuto la bussata sulla porta 23 (Telnet) e l'ha chiusa immediatamente — perché `transport input ssh` gli impedisce di accettarla.
>
> In altre parole: il router risponde, ma rifiuta il protocollo. La sicurezza funziona.

> [!CAUTION]
> Se invece Telnet si apre normalmente e ti chiede la password, la configurazione è sbagliata. Verifica con:
>
> ```
> RTR-A# show running-config | section vty
> ```
>
> Devi vedere `transport input ssh` — se vedi `transport input telnet` o nulla, torna alla Parte 2B e riapplica il comando.

## Parte 4 — Mettere in Sicurezza gli Switch (SW-1 e SW-2)
 
Sugli switch la logica di sicurezza è identica a quella del router, ma con una differenza fondamentale: lo switch **non ha un indirizzo IP per default**.
 
### Perché lo switch non ha un IP?
 
Uno switch di Livello 2 appena uscito dalla scatola lavora leggendo **indirizzi MAC** — non ha bisogno di sapere nulla di IP per smistare il traffico tra le porte. Potresti lasciarlo senza IP per anni e continuerebbe a funzionare perfettamente.
 
> [!NOTE]
> Il router è diverso: ogni sua porta fisica è il confine tra due reti, quindi deve avere un IP reale su ciascuna interfaccia. Lo switch invece è "trasparente" — il traffico ci passa attraverso senza che lui debba partecipare.
 
### L'IP virtuale: cos'è la SVI (Switch Virtual Interface)?
 
Le porte fisiche di uno switch non accettano un indirizzo IP — se provi a scriverlo su `interface Fa0/1` riceverai un errore. La soluzione è creare un'**interfaccia logica**: la **VLAN 1**.
 
Assegnando l'IP alla VLAN 1, lo switch diventa un "host" nella rete esattamente come un PC. Da quel momento è raggiungibile via SSH, risponde al ping e può inviare avvisi di monitoraggio.
 
> [!IMPORTANT]
> Questa è la differenza tra **usare** la rete e **amministrarla**. Senza IP lo switch smista traffico ma è cieco e sordo per te. Con l'IP diventa gestibile — e proprio per questo va protetto esattamente come un router.
 
Ripeti i passaggi seguenti su entrambi gli switch (IP `.2` per SW-1, IP `.3` per SW-2).
 
---
 
### 1. Configurazione Identità e IP di Gestione
 
```bash
S1> enable
S1# configure terminal
S1(config)# hostname SW-1
S1(config)# interface vlan 1
S1(config-if)# ip address 192.168.1.2 255.255.255.0
S1(config-if)# no shutdown
S1(config-if)# exit
S1(config)# ip default-gateway 192.168.1.1
```
 
> [!NOTE]
> Il comando `ip default-gateway` è indispensabile sullo switch: senza di esso lo switch può ricevere connessioni SSH solo da PC sulla stessa rete locale, ma non sa come rispondere a pacchetti che arrivano da altre reti attraverso il router.
 
---
 
### 2. Attivazione SSH e Sicurezza Account
 
La procedura è identica a quella del router — lo switch diventa un host sulla rete e va blindato allo stesso modo.
 
```bash
# Password di sistema e crittografia
SW-1(config)# enable secret @Sicurezza123!
SW-1(config)# service password-encryption
 
# Dominio e utente per SSH
SW-1(config)# ip domain-name lab.sicuro.it
SW-1(config)# username SSHadmin secret @AdminPass123!
SW-1(config)# crypto key generate rsa
```
 
> Quando richiesto, inserisci: `1024`
 
```bash
# Linee virtuali: solo SSH, autenticazione locale
SW-1(config)# line vty 0 15
SW-1(config-line)# transport input ssh
SW-1(config-line)# login local
SW-1(config-line)# exit
```
 
> [!TIP]
> Sullo switch si usano `line vty 0 15` (16 sessioni) invece di `0 4` come sul router — gli switch supportano più sessioni simultanee per default. La logica di sicurezza rimane identica.
 
---
 
### 3. Sicurezza Fisica: Shutdown delle Porte Inutilizzate
 
```bash
SW-1(config)# interface range fa0/4 - 24
SW-1(config-if-range)# shutdown
SW-1(config-if-range)# exit
```
 
> [!TIP]
> Spegnere le porte inutilizzate impedisce a chiunque di collegare un dispositivo non autorizzato semplicemente inserendo un cavo. Controlla il tuo schema di rete per verificare quali porte sono effettivamente in uso prima di eseguire questo comando.

## Parte 5 — Test di Connessione e Verifica

Dopo aver configurato tutto, è fondamentale verificare che la "blindatura" funzioni correttamente da diversi punti della rete.

### Test 1: Dal PC dell'Amministratore
**Nota bene:** Assicurati che il PC abbia un IP univoco (es. `192.168.1.10`) per evitare conflitti con lo switch.

1.  Apri il **Command Prompt** sul PC.
2.  Esegui un ping per testare la connettività: `ping 192.168.1.2`
3.  Accedi via SSH:
    ```bash
    C:\> ssh -l SSHadmin 192.168.1.2
    # Inserisci la password: @AdminPass123!
    ```

### Test 2: Dal Router (RTR-A)
Il router può essere usato come punto di gestione centrale per saltare da un dispositivo all'altro.

1.  Entra nella console del Router.
2.  Prova a collegarti allo switch:
    ```bash
    RTR-A# ssh -l SSHadmin 192.168.1.2
    # Se ti chiede la password, il tunnel è attivo e sicuro.
    ```

### Test 3: Verifica dello stato SSH sullo Switch
Per essere sicuri che lo switch stia usando i protocolli corretti, usa questo comando sulla console dello switch:
```bash
SW-1# show ip ssh
# Deve apparire: "SSH Enabled - version 1.99" (o 2.0)
```

> [!CAUTION]
> **Promemoria finale:** Una volta terminati i test, salva la configurazione per renderla permanente con il comando `copy running-config startup-config` su ogni dispositivo. Se non lo fai, al riavvio perderai tutta la blindatura!

---

## Parte 6 — Salvataggio Finale 💾

```bash
RTR-A# copy running-config startup-config
```

> [!IMPORTANT]
> Senza questo comando, **tutta la configurazione viene persa al riavvio**. La `running-config` vive solo in RAM. La `startup-config` è quella che il router carica all'accensione dalla memoria permanente (NVRAM).

## 📝 Troubleshooting Rapido

| Problema | Causa probabile | Soluzione |
|---|---|---|
| SSH rifiuta la connessione | Il PC non ha un IP configurato | Imposta IP `192.168.1.10`, subnet `255.255.255.0`, gateway `192.168.1.1` |
| "Connection refused" immediato | Sei nel periodo di blocco brute force | Aspetta 120 secondi e riprova |
| Telnet funziona ancora | `transport input ssh` mancante o su linee sbagliate | Verifica con `show running-config \| section vty` |
| Le chiavi RSA non si generano | `ip domain-name` non configurato | Aggiungi `ip domain-name lab.sicuro.it` e riprova |
