# Modulo 17.6 – Risoluzione dei Problemi di Rete (Troubleshooting)

---

## 17.6.1 Approccio Base alla Risoluzione dei Problemi

Nei moduli precedenti sono state introdotte utilità e comandi utili per identificare le aree problematiche della rete. La risoluzione dei problemi è una competenza fondamentale per qualsiasi amministratore di rete.

I problemi di rete possono essere semplici o complessi e possono derivare da una combinazione di problemi di **hardware**, **software** e **connettività**. Prima di poter risolvere un problema, il tecnico deve saper analizzare la situazione e determinare la causa dell'errore: questo processo si chiama **troubleshooting**.

Una metodologia comune ed efficiente si basa sul **metodo scientifico** ed è articolata in **sei fasi principali**:

---

### Fasi del Processo di Troubleshooting

| Fase | Nome | Descrizione |
|------|------|-------------|
| **1** | Identificare il Problema | Prima fase del processo. Oltre agli strumenti diagnostici, una conversazione diretta con l'utente è spesso molto utile. |
| **2** | Stabilire una teoria di probabili cause | Dopo aver identificato il problema, si formulano più ipotesi sulle possibili cause. |
| **3** | Verificare la teoria sulle probabili cause | Le teorie vengono testate per individuare la causa effettiva. Se una procedura rapida non risolve il problema, si approfondisce l'analisi. |
| **4** | Definire un piano d'azione e implementare la soluzione | Determinata la causa esatta, si pianifica e si applica la soluzione. |
| **5** | Verificare la soluzione e implementare misure preventive | Si verifica la piena funzionalità del sistema e si applicano, se possibile, misure preventive. |
| **6** | Conclusioni, Azioni e Risultati | Si documentano le conclusioni, le azioni intraprese e i risultati ottenuti. Questa fase è fondamentale come riferimento futuro. |

---

> [!TIP]
> ### 💡 Suggerimento operativo
> Per valutare correttamente il problema, stabilisci **quanti dispositivi** nella rete sono coinvolti:
> - Se il problema riguarda **un solo dispositivo**, avvia il troubleshooting da quel dispositivo.
> - Se il problema riguarda **tutti i dispositivi**, avvia il troubleshooting dal dispositivo al quale sono connessi tutti gli altri (es. switch o router centrale).
>
> Sviluppa sempre un metodo **logico e coerente**, eliminando un problema alla volta.

## 17.6.2 Risolvere il Problema o Riassegnarlo?

In alcuni casi, la soluzione immediata di un problema non è fattibile da parte del tecnico incaricato. In questi casi è necessario procedere con la **riassegnazione (escalation)** del problema.

La riassegnazione è necessaria quando:

- È richiesta la **decisione di un manager**.
- Sono necessarie **competenze specifiche** non disponibili al tecnico.
- Il tecnico non dispone del **livello di accesso alla rete** necessario.

### Esempio pratico

> Dopo la risoluzione dei problemi, il tecnico conclude che è necessario **sostituire il modulo di un router**. In questo caso:
> 1. Il tecnico riassegna il problema al **manager** per ottenere l'approvazione.
> 2. Il manager potrebbe a sua volta riassegnare al **reparto contabile** per approvare l'acquisto del nuovo modulo.

> [!IMPORTANT]
> ### 📋 Policy aziendale
> Il criterio e le modalità di riassegnazione di un problema da parte di un tecnico devono essere **specificati chiaramente nella policy aziendale**.

## 17.6.3 Il Comando `debug`

I processi del sistema operativo, i protocolli, i meccanismi e gli eventi generano **messaggi di stato** che possono fornire informazioni preziose per la risoluzione dei problemi.

Il comando **`debug`** di Cisco IOS consente all'amministratore di visualizzare questi messaggi **in tempo reale**. È uno strumento fondamentale per il monitoraggio degli eventi su un dispositivo Cisco IOS.

> [!CAUTION]
> ### ⚠️ Attenzione all'utilizzo del debug
> All'output del comando `debug` viene assegnata **alta priorità** nel processo CPU. Un utilizzo eccessivo può rendere il sistema inutilizzabile, poiché il router sarebbe troppo occupato a visualizzare i messaggi per eseguire le normali funzioni di rete.
> - **Evitare** `debug all` e `debug ip packet` in ambienti di produzione.
> - Utilizzare il debug **solo per risolvere problemi specifici**.
> - **Disabilitare sempre** il debug non appena non è più necessario.

### Modalità di esecuzione

Tutti i comandi `debug` vengono eseguiti in **modalità Privileged EXEC**.

### Esempio: monitoraggio dei messaggi ICMP

```

ICMP packet debugging is on

R1# ping 10.1.1.1

!!!!!

*Aug 20 14:18:59.605: ICMP: echo reply rcvd, src 10.1.1.1, dst 209.165.200.225, topology BASE, dscp 0
*Aug 20 14:18:59.606: ICMP: echo reply rcvd, src 10.1.1.1, dst 209.165.200.225, topology BASE, dscp 0
*Aug 20 14:18:59.608: ICMP: echo reply rcvd, src 10.1.1.1, dst 209.165.200.225, topology BASE, dscp 0
*Aug 20 14:18:59.609: ICMP: echo reply rcvd, src 10.1.1.1, dst 209.165.200.225, topology BASE, dscp 0
*Aug 20 14:18:59.611: ICMP: echo reply rcvd, src 10.1.1.1, dst 209.165.200.225, topology BASE, dscp 0
```

### Comandi principali

| Comando | Descrizione |
|---------|-------------|
| `debug ip icmp` | Attiva il debug per i messaggi ICMP |
| `debug ?` | Mostra tutte le opzioni disponibili del comando debug |
| `no debug ip icmp` | Disattiva il debug ICMP specifico |
| `undebug ip icmp` | Forma alternativa per disattivare il debug ICMP |
| `undebug all` | Disattiva **tutti** i comandi debug attivi contemporaneamente |

```
Router# no debug ip icmp
Router# undebug ip icmp
Router# undebug all
```

> [!NOTE]
> ### 📝 Nota
> L'intento del comando `debug` è catturare output **dal vivo per un breve periodo di tempo** (da pochi secondi a circa un minuto). Disabilita sempre il debug non appena non è più necessario.

## 17.6.4 Il Comando `terminal monitor`

Le connessioni all'interfaccia CLI (Command Line Interface) di Cisco IOS possono essere stabilite in due modalità:

- **Localmente** — Connessione fisica alla **porta console** del router o switch tramite un cavo di rollover.
- **In remoto** — Connessione tramite **Telnet** o **SSH** a un dispositivo con IP configurato.

### Problema con le connessioni remote

Per impostazione predefinita, l'output del comando `debug` viene visualizzato **solo sulle connessioni console** e **non** sulle connessioni remote (Telnet/SSH), perché i messaggi di log vengono bloccati sulle linee VTY.

### Esempio: debug non visibile su connessione Telnet

```
R2# telnet 209.165.200.225
Trying 209.165.200.225 ... Open
Authorized access only!

R1> enable
R1# debug ip icmp
ICMP packet debugging is on

R1# ping 10.1.1.1
!!!!!
Success rate is 100 percent (5/5), round-trip min/avg/max = 1/1/2 ms

! Nessun output debug visualizzato
```

### Soluzione: il comando `terminal monitor`

Per visualizzare i messaggi di log su un terminale remoto (console virtuale), utilizzare il comando:

```
R1# terminal monitor
```

Per interrompere la visualizzazione:

```
R1# terminal no monitor
```

```

R1# ping 10.1.1.1
```

## Domande di Verifica

---

### Domanda 1

**Un tecnico sta risolvendo un problema di rete e ha appena stabilito una teoria delle cause probabili. Quale sarebbe il passo successivo nel processo di troubleshooting?**

- Verificare la soluzione e implementare misure preventive
- Definire un piano d'azione e implementare la soluzione
- ✅ **Verificare la teoria per individuare la causa**
- Identificare il problema
- Documentare le conclusioni, le azioni e i risultati

#### Spiegazione

La risposta corretta è **"Verificare la teoria per individuare la causa"** (Fase 3).

Il processo di troubleshooting segue un ordine preciso:

| Fase | Azione |
|------|--------|
| 1 | Identificare il problema |
| 2 | Stabilire una teoria di cause probabili ← *il tecnico è qui* |
| **3** | **Verificare la teoria ← passo successivo** |
| 4 | Definire un piano d'azione e implementare la soluzione |
| 5 | Verificare la soluzione e misure preventive |
| 6 | Documentare conclusioni, azioni, risultati |

Le altre opzioni sono **errate** perché appartengono a fasi successive (4, 5, 6) o precedenti (1), non al passo immediatamente successivo alla Fase 2.

---

### Domanda 2

**Un tecnico sta risolvendo un problema di rete. Dopo la risoluzione dei problemi, il tecnico conclude che è necessario sostituire uno switch. Cosa dovrebbe fare successivamente il tecnico?**

- Risolvere il problema.
- ✅ **Riassegnare il ticket di assistenza al manager per approvare la modifica.**
- Acquistare un nuovo switch e sostituire quello difettoso.
- Inviare un'e-mail a tutti gli utenti per far sapere che sta sostituendo uno switch.

#### Spiegazione

La risposta corretta è **"Riassegnare il ticket al manager"**.

Sostituire un componente hardware implica:
- Un'**approvazione manageriale** (decisione aziendale).
- Un possibile **acquisto** che richiede autorizzazione del reparto contabile.

Il tecnico **non può agire autonomamente** in questi casi. Le altre opzioni sono errate perché:
- **Risolvere il problema** non è possibile senza prima avere approvazione e il componente.
- **Acquistare direttamente** viola le procedure aziendali di approvazione.
- **Notificare gli utenti via e-mail** non è una procedura standard né il passo successivo corretto in questa fase.

---

### Domanda 3

**Un tecnico utilizza il comando `debug ip icmp` per acquisire l'output in tempo reale. Quali comandi interromperebbero questo debug? (Scegliere due risposte.)**

- debug ip icmp off
- ✅ **undebug all**
- undebug debug ip icmp
- ✅ **no debug ip icmp**
- no debug debug ip icmp

#### Spiegazione

Le due risposte corrette sono:

- **`no debug ip icmp`** — disattiva specificamente il debug ICMP.
- **`undebug all`** — disattiva **tutti** i comandi debug attivi contemporaneamente.

Le altre opzioni sono **errate** perché:
- **`debug ip icmp off`** — la sintassi `off` non è valida in Cisco IOS.
- **`undebug debug ip icmp`** — sintassi non valida (doppio `debug`).
- **`no debug debug ip icmp`** — sintassi non valida (doppio `debug`).

> [!NOTE]
> In alternativa a `no debug ip icmp`, è possibile usare anche `undebug ip icmp`, che è la forma equivalente accettata da Cisco IOS.

---

### Domanda 4

**Un tecnico ha stabilito una connessione remota al router R1 per osservare l'output debug. Il tecnico inserisce `debug ip icmp`, poi esegue un ping, ma non viene visualizzato alcun output. Quale comando deve immettere?**

- ✅ **terminal monitor**
- monitor terminal
- terminal monitor debug
- monitor debug output

#### Spiegazione

La risposta corretta è **`terminal monitor`**.

Per impostazione predefinita, l'output del comando `debug` **non viene inviato alle sessioni remote** (Telnet/SSH) perché i messaggi di log sono bloccati sulle linee VTY. Il comando `terminal monitor` abilita la visualizzazione dei messaggi di log sul terminale remoto corrente.

Le altre opzioni sono **errate** perché:
- **`monitor terminal`** — ordine dei termini invertito, non è un comando IOS valido.
- **`terminal monitor debug`** — sintassi non valida.
- **`monitor debug output`** — sintassi non valida, non esiste in Cisco IOS.

Per disattivare la funzione, si usa `terminal no monitor`.

## Glossario / Ripasso Rapido

| Termine / Comando | Descrizione |
|-------------------|-------------|
| **Troubleshooting** | Processo strutturato di identificazione e risoluzione dei problemi di rete |
| **Escalation (Riassegnazione)** | Trasferimento di un problema a un livello superiore (manager, specialista) quando il tecnico non può risolverlo autonomamente |
| **Privileged EXEC** | Modalità di accesso avanzato alla CLI Cisco IOS (prompt: `Router#`), necessaria per comandi di configurazione e debug |
| **`debug`** | Comando IOS per visualizzare messaggi di sistema in tempo reale; da usare con cautela |
| **`debug ip icmp`** | Attiva il debug per i pacchetti ICMP su un router Cisco |
| **`no debug ip icmp`** | Disattiva il debug ICMP specifico |
| **`undebug ip icmp`** | Forma alternativa di `no debug ip icmp` |
| **`undebug all`** | Disattiva tutti i comandi debug attivi contemporaneamente |
| **`debug ?`** | Visualizza tutte le opzioni disponibili del comando debug in modalità Privileged EXEC |
| **`terminal monitor`** | Abilita la visualizzazione dei messaggi di log (incluso l'output debug) su sessioni remote (Telnet/SSH) |
| **`terminal no monitor`** | Disabilita la visualizzazione dei messaggi di log su sessioni remote |
| **Console** | Connessione locale fisica alla porta console del dispositivo tramite cavo di rollover |
| **VTY (Virtual Terminal Lines)** | Linee virtuali utilizzate per le connessioni remote (Telnet/SSH) a dispositivi Cisco |
| **Telnet** | Protocollo per la gestione remota di dispositivi di rete (non cifrato) |
| **SSH (Secure Shell)** | Protocollo per la gestione remota sicura (cifrata) di dispositivi di rete |
| **ICMP** | Internet Control Message Protocol — usato per messaggi di controllo e diagnostica di rete (es. `ping`) |
| **IOS (Internetwork Operating System)** | Sistema operativo proprietario Cisco per router e switch |
| **CLI (Command Line Interface)** | Interfaccia a riga di comando per la gestione di dispositivi Cisco IOS |
| **Cavo di Rollover** | Cavo speciale usato per collegare un PC alla porta console di un router o switch Cisco |
| **CPU Priority** | Il debug ha alta priorità CPU: un uso eccessivo può rendere il dispositivo inutilizzabile |
